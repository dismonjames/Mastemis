# Problem authoring backend

The application layer exposes authorized draft creation, MAS validation, bounded preview, generation status, cancellation, and atomic publication. Preview never publishes tests and returns the seed, runtime version, diagnostics, bounded inputs, and truncation state.

PostgreSQL mode implements `IProblemStudioStore`. The `AddProblemAuthoring` migration creates draft, localized statement metadata, generation operation, generated test set, generated test, and package import/export audit tables. Drafts and operations have optimistic concurrency tokens. A partial unique index permits one pending/running generation per draft, one test set is allowed per operation, and test indexes are unique within a set.

The authenticated `/api/problem-studio` surface supports draft creation/listing/retrieval/update/deletion, author and examination assignments, localized statements, bounded assets, revisioned MAS retrieval/update/validation/preview, reference-solution revisions, generation start/status/progress/diagnostics/cancellation, published-test inspection, package validation, CreateNew and ReplaceDraft import, import status, and deterministic retained exports. Cookie-authenticated requests on this surface require an antiforgery token. Owner and Editor assignments may mutate; Reviewer and Viewer assignments are read-only. ExamManager access is scoped through examination assignments. Candidates, room invigilators, evidence reviewers, and judge-worker credentials do not receive authoring access. Open examinations block mutation and publication.

Large authoring content uses generated object identifiers. Files are written atomically beneath a canonical storage root, bounded and SHA-256 hashed, and remain staged until relational publication commits. Reconciliation asks PostgreSQL which staged identifiers are referenced; referenced staged files are promoted and only stale unreferenced files are removed. A database lookup failure aborts cleanup.

Reference-output work is a separate durable PostgreSQL queue. Authenticated workers claim it with `FOR UPDATE SKIP LOCKED`, renew an unpredictable lease, download hash-verified sources and staged inputs, compile once, run every test through the fail-closed sandbox, and upload bounded hash-addressed outputs. Publication verifies complete input/output metadata and commits one published set, judge profile, operation state, and outbox message atomically; object promotion follows the relational commit and is recoverable by reconciliation.

CreateNew package import is durable and idempotent by request key. ReplaceDraft is a separate optimistic-concurrency workflow: it requires the package problem identifier to equal the target, rejects open-examination mutation, stages every object before one relational replacement transaction, and retains immutable import and revision history. A retry with the same successful key returns the prior import; a conflicting key use is rejected.

Export requires explicit hidden-test permission and uses deterministic entry order, fixed ZIP timestamps, stable JSON, and checksums. Package bytes are written under a generated object identifier. An append-only export row records actor, problem version, hidden-content scope, hash, length, format, status, and retention deadline. The row moves from `Staged` to `Ready` only after object promotion. `Failed` and expired objects are cleanup candidates; downloads require a `Ready`, unexpired row and never expose the physical path.

Normal submission jobs and reference-output jobs share one in-process worker capacity coordinator. Reference-output claimers wait for the next shared slot while submission claimers use currently available slots, preventing combined overcommit and giving a waiting authoring job a bounded opportunity when a slot is released. Durable PostgreSQL leases remain authoritative after worker failure.
